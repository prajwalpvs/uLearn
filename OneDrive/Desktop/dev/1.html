Two Sum
Easy
Hash Map · Arrays
Given an array of integers and a target, return indices of two numbers that add up to the target.
► show solution
def two_sum(nums, target):
    seen = {}  # value -> index
    for i, n in enumerate(nums):
        complement = target - n
        if complement in seen:
            return [seen[complement], i]
        seen[n] = i
    return []
Time: O(n)
Space: O(n)
Valid Parentheses
Easy
Stack · Strings
Determine if a string of brackets ()[]{} is valid (properly opened and closed).
► show solution
def is_valid(s):
    stack = []
    pairs = {')': '(', ']': '[', '}': '{'}
    for ch in s:
        if ch in '([{':
            stack.append(ch)
        else:
            if not stack or stack.pop() != pairs[ch]:
                return False
    return not stack
Time: O(n)
Space: O(n)
Reverse Linked List
Easy
Linked List · Pointers
Reverse a singly linked list iteratively in-place.
► show solution
def reverse(head):
    prev = None
    curr = head
    while curr:
        nxt = curr.next
        curr.next = prev
        prev = curr
        curr = nxt
    return prev
Time: O(n)
Space: O(1)
Best Time to Buy/Sell Stock
Easy
Greedy · Arrays
Given prices array, find max profit from one buy + one sell.
► show solution
def max_profit(prices):
    min_price = float('inf')
    profit = 0
    for p in prices:
        min_price = min(min_price, p)
        profit = max(profit, p - min_price)
    return profit
Time: O(n)
Space: O(1)
Longest Substring No Repeat
Medium
Sliding Window · Hash Set
Find the length of the longest substring with no repeating characters.
► show solution
def length_of_longest(s):
    seen = {}
    left = max_len = 0
    for right, ch in enumerate(s):
        if ch in seen and seen[ch] >= left:
            left = seen[ch] + 1
        seen[ch] = right
        max_len = max(max_len, right - left + 1)
    return max_len
Time: O(n)
Space: O(min(n, charset))
LRU Cache
Medium
Hash Map · Doubly Linked List
Implement Least Recently Used cache with O(1) get and put operations.
► show solution
from collections import OrderedDict

class LRUCache:
    def __init__(self, cap):
        self.cap = cap
        self.cache = OrderedDict()

    def get(self, key):
        if key not in self.cache:
            return -1
        self.cache.move_to_end(key)
        return self.cache[key]

    def put(self, key, val):
        if key in self.cache:
            self.cache.move_to_end(key)
        self.cache[key] = val
        if len(self.cache) > self.cap:
            self.cache.popitem(last=False)
Time: O(1) get/put
Space: O(capacity)
Number of Islands
Medium
DFS / BFS · Grid
Count islands in a 2D grid of '1's (land) and '0's (water).
► show solution
def num_islands(grid):
    if not grid: return 0
    rows, cols = len(grid), len(grid[0])
    count = 0

    def dfs(r, c):
        if r < 0 or c < 0 or r >= rows \
           or c >= cols or grid[r][c] != '1':
            return
        grid[r][c] = '0'

TCP vs UDP

TCP — Transmission Control Protocol

  • Connection-oriented (SYN → SYN-ACK → ACK handshake)
  • Guaranteed delivery, in-order, error-checked
  • Flow control + congestion control built in
  • Higher overhead — use when reliability matters
  • Use: HTTP, SSH, databases, email (SMTP)

UDP — User Datagram Protocol

  • Connectionless — fire and forget, no handshake
  • No delivery guarantee, no ordering
  • Very low overhead and latency
  • App handles packet loss if needed
  • Use: DNS, video streaming, gaming, VoIP, HTTP/3 (QUIC)

  • DoH (DNS over HTTPS) and DoT (DNS over TLS) encrypt DNS queries to prevent eavesdropping
HTTP Evolution
  • HTTP/1.1 — persistent connections but HOL blocking; effectively 1 in-flight request per connection despite pipelining
  • HTTP/2 — binary framing; multiplexing (many streams per TCP conn); header compression (HPACK); server push
  • HTTP/3 — runs on QUIC (UDP-based); eliminates TCP HOL blocking; 0-RTT session resumption; built-in TLS 1.3; better on lossy networks
TLS 1.3 Handshake
  • ClientHello — supported ciphers, DH key share, random nonce
  • ServerHello — chosen cipher, server key share, certificate chain
  • Client verifies cert against trusted CA roots in browser/OS store
  • Session keys derived via Diffie-Hellman — secret never transmitted
  • TLS 1.3: 1 RTT total (vs 2 for TLS 1.2); 0-RTT for session resumption
  • PFS: ephemeral keys mean past traffic can't be decrypted if private key leaks later
WebSockets vs SSE vs Polling
  • WebSocket — full-duplex, persistent; upgrades from HTTP via Upgrade: websocket; use for chat, games, live collab
  • SSE (Server-Sent Events) — server-to-client only; uses plain HTTP; simpler; reconnects automatically; use for dashboards, notifications
  • Long polling — client holds request open; server replies when data ready; fallback for restricted environments
  • Scale WebSockets horizontally with sticky sessions or shared Redis pub/sub
Load Balancers: L4 vs L7
  • L4 (Transport) — routes by IP + TCP/UDP port; no content inspection; very fast; e.g. AWS NLB, HAProxy TCP
  • L7 (Application) — routes by URL path, headers, cookies; SSL termination; A/B routing; e.g. AWS ALB, nginx, Envoy
  • Algorithms: Round Robin, Least Connections, IP Hash (sticky), Weighted RR
  • Health checks: periodic probes remove unhealthy backends automatically
  • CDN (Cloudflare, CloudFront) = globally distributed L7 cache + anycast edge routing
Authentication vs Authorization

Authentication (AuthN) — Who are you?

  • Prove identity: password, biometric, hardware key, OTP
  • Result: session cookie, JWT, signed token
  • Protocols: OAuth 2.0, OIDC, SAML, LDAP
  • Factors: something you know / have / are → MFA

Authorization (AuthZ) — What can you do?

  • Checks permissions after identity confirmed
  • RBAC — roles (admin, editor, viewer)
  • ABAC — attributes (dept=eng, clearance=3)
  • Principle of Least Privilege — grant minimum needed
  • Tools: OPA (Open Policy Agent), Casbin, AWS IAM

// Payload — claims (public + private)
{ "sub": "user_123", "role": "admin", "exp": 1735689600 }

// Signature — server verifies on every request
RSASHA256(base64url(header) + "." + base64url(payload), privateKey)
  • Stateless — server validates signature without DB lookup (ideal for microservices)
  • Prefer RS256 (asymmetric) over HS256 (shared secret) for distributed systems
  • Short expiry (15 min access token) + long-lived refresh token + rotation = revocability
  • Store in HttpOnly cookie, NOT localStorage — prevents XSS token theft
  • Payload is base64-encoded, not encrypted — never put secrets in JWT claims
OAuth 2.0 Grant Types
Authorization Code + PKCE (web / mobile)
  • User authenticates at AuthServer → redirected back with a one-time code
  • Client exchanges code + code_verifier (PKCE) for access + refresh tokens
  • PKCE prevents code interception attacks — safe for public clients (SPA, mobile)
  • Never expose client_secret in frontend JS — use PKCE instead
Client Credentials (Machine-to-Machine)
  • Service sends client_id + client_secret directly to AuthServer for token
  • No user involved — for backend-to-backend API calls
  • Rotate secrets via secrets manager; never hardcode in source code
  • Scope access narrowly — one client per service per environment
Password Hashing
Use bcrypt / Argon2id / scrypt — never raw MD5, SHA1, or SHA256
import bcrypt

# Hash on registration (cost=12 ≈ 300ms — slow is the point)
hashed = bcrypt.hashpw(password.encode(), bcrypt.gensalt(rounds=12))

# Verify on login — uses constant-time comparison internally
is_valid = bcrypt.checkpw(candidate.encode(), hashed)
  • bcrypt automatically salts each hash — no separate salt management needed
  • Argon2id is the 2023 OWASP recommendation — memory-hard, resistant to GPU cracking
  • MD5 / SHA-256 are fast checksums for data integrity, not designed for passwords
  • Increase cost factor over time as hardware improves; re-hash on next login

SQL Injection
# Vulnerable — attacker passes id="1 OR 1=1"
query = f"SELECT * FROM users WHERE id={user_id}"

# Safe — parameterized query

Use ORMs or parameterized queries everywhere. Grant DB accounts only the permissions they need. Validate and whitelist all user input at the boundary.

XSS — Cross-Site Scripting
// Vulnerable — injects attacker JS into DOM
element.innerHTML = userInput;

// Safe — treats as plain text
element.textContent = userInput;
// Rich text: sanitize with DOMPurify first

Escape all user-controlled output in templates. Set Content-Security-Policy header. Use HttpOnly cookies so JS cannot read auth tokens even if XSS occurs.

CSRF — Cross-Site Request Forgery
  • Attacker's page tricks logged-in user's browser into sending a forged request to your API
  • Defense 1: CSRF token — random, per-session, validated server-side on every mutating request
  • Defense 2: SameSite=Strict or SameSite=Lax cookie attribute blocks cross-origin sends
  • Defense 3: Verify Origin / Referer headers on state-changing endpoints
Cookie Security Flags + CORS
  • HttpOnly — JS cannot read cookie; blocks XSS token theft
  • Secure — only sent over HTTPS; never over HTTP
  • SameSite=Strict — never sent cross-site (may break OAuth redirects)
  • SameSite=Lax — sent on top-level navigation only (safe default)
  • CORS: Access-Control-Allow-Origin restricts which origins can call your API; never use * with credentials

Serverless vs Containers

Serverless (Lambda / Cloud Functions)

  • Zero server management; auto-scales to millions of requests
  • Pay per invocation + duration (ms granularity)
  • Cold start: 100ms–2s on first invocation (use Provisioned Concurrency to eliminate)
  • Max execution: 15 min (Lambda); no persistent connections
  • Best for: event handlers, webhooks, scheduled jobs, glue code, bursty workloads

Containers (ECS / GKE / AKS)

  • Full control over runtime, dependencies, OS packages
  • Predictable latency; no cold starts; supports persistent connections
  • Pay per running instance even when idle
  • Best for: long-running APIs, WebSocket servers, ML inference, high-throughput services
  • Ops overhead: health checks, autoscaling config, rolling deploys, image scanning

A
ACID — Atomicity, Consistency, Isolation, Durability: the four guarantees a relational database transaction must satisfy.
Abstraction — Hiding implementation complexity behind a clean interface so callers only see what they need.
Agile — Iterative delivery methodology built around short sprints, continuous feedback, and working software over documentation.
API — Application Programming Interface; a defined contract specifying how two systems communicate.
Asynchronous — Operations that return immediately; the caller handles the result later via callback, promise, or event.
Atomicity — A transaction either fully completes or fully rolls back; no partial intermediate state is ever left visible.
B
Backpressure — A flow-control signal where a downstream consumer tells an upstream producer to slow down.
Big-O Notation — Describes the worst-case growth rate of an algorithm's time or space as input size n grows.
Bloom Filter — A probabilistic data structure that answers "possibly in set" or "definitely not in set" using minimal memory.
Blue-Green Deployment — Two identical production environments; deploy to the idle one, then switch traffic instantly for zero downtime.
Branching Strategy — A convention for creating and merging code branches, e.g. Git Flow or trunk-based development.
C
CAP Theorem — In a distributed system you can guarantee at most two of: Consistency, Availability, Partition Tolerance.
Cache Invalidation — Removing or updating stale cached data; notoriously one of the two hard problems in computer science.
Canary Release — Rolling a new version out to a small fraction of users first to catch problems before full deployment.
CDN — Content Delivery Network; globally distributed servers that cache static assets close to end users.
Circuit Breaker — Stops forwarding requests to a failing dependency, failing fast and allowing the dependency time to recover.
Consistency — Every read sees the most recent committed write or returns an error (versus eventual consistency).
Container — Lightweight portable runtime package bundling app code and dependencies; isolated from the OS but sharing the kernel.
CQRS — Command Query Responsibility Segregation; using separate models for reads and writes to scale them independently.
CRUD — Create, Read, Update, Delete: the four fundamental operations on persistent data.
D
Dead Letter Queue — A queue where unprocessable messages land after repeated delivery failures, for later inspection.
Deadlock — Two or more processes each waiting for a lock held by the other; none can proceed.
Declarative — You describe the desired end-state, not the steps to reach it. SQL and Kubernetes manifests are declarative.
Dependency Injection — Passing a component's dependencies from outside rather than having it construct them internally.
DevOps — Culture and practice uniting development and operations to shorten release cycles and improve reliability.
DNS — Domain Name System; translates human-readable hostnames to IP addresses via a distributed hierarchy.
E
Encapsulation — Bundling data with the methods that operate on it and restricting direct access to internals.
Event Sourcing — Storing state as an append-only log of events; current state is reconstructed by replaying them.
Eventual Consistency — Replicas will converge to the same value given sufficient time and the absence of new writes.
Exponential Backoff — Retry strategy where the wait time doubles after each failure to reduce thundering-herd load.
F
Feature Flag — A runtime switch enabling or disabling a feature without a code deployment. Also called feature toggle.
FIFO — First In, First Out: the ordering discipline of a queue data structure.
Functional Programming — A paradigm treating computation as pure function evaluation; avoids shared mutable state.
G
Garbage Collection — Automated runtime reclamation of memory held by objects no longer reachable from any reference.
gRPC — Google's high-performance RPC framework using Protocol Buffers over HTTP/2 for efficient binary communication.
H
Hash Function — Maps arbitrary input to a fixed-size output deterministically; identical inputs always produce identical outputs.
Horizontal Scaling — Adding more machines to distribute load (scale out). Contrasts with vertical scaling (scale up).
Hot Spot — A shard or partition receiving disproportionately high traffic, creating a localized performance bottleneck.
HTTP/2 — HTTP revision enabling request multiplexing over one TCP connection, header compression, and server push.
Hysteresis — Intentional lag before reacting to a state change, preventing rapid oscillation, e.g. autoscaling jitter.
I
Idempotency — An operation that produces the same result whether applied once or many times. Critical for safe retries.
Immutability — Data that cannot be changed after creation; updates produce new values rather than modifying existing ones.
Indexing — An auxiliary data structure (B-tree, hash) letting a DB locate rows without scanning the full table.
Infrastructure as Code — Managing infrastructure via version-controlled config files (Terraform, CloudFormation) instead of manual steps.
J
JWT — JSON Web Token; a signed, self-contained token encoding claims, widely used for stateless authentication.
K
Kubernetes — Container orchestration platform automating deployment, scaling, and management of containerized workloads.
L
Latency — Time elapsed for a single request to travel end-to-end. Measured in ms; distinct from throughput.
LIFO — Last In, First Out: the ordering discipline of a stack data structure.
Load Balancer — Distributes incoming requests across multiple backend servers to prevent any single one being overwhelmed.
LoRA — Low-Rank Adaptation; efficient LLM fine-tuning technique inserting small trainable matrices into frozen model weights.
LRU Cache — Least Recently Used; eviction policy that removes the item accessed least recently when the cache is full.
M
Marshalling — Converting an in-memory object into a transmittable format (JSON, Protobuf). Also called serialization.
Memoization — Caching function results by input so identical calls return the cached value instead of recomputing.
Message Queue — Async communication buffer where producers publish messages and consumers retrieve and process them independently.
Microservices — Architectural style decomposing an application into small, independently deployable services.
Monolith — A single deployable unit containing all application components; simpler to develop, harder to scale independently.
Mutex — Mutual-exclusion lock ensuring only one thread accesses a shared resource at a time.
N
N+1 Problem — Fetching N records then issuing one extra query per record; causes exponential database load.
Normalization — Organizing relational tables to reduce redundancy and improve data integrity (1NF, 2NF, 3NF, BCNF).
NoSQL — Non-relational databases: document (MongoDB), key-value (Redis), column-family (Cassandra), graph (Neo4j).
O
OAuth 2.0 — Authorization framework letting third-party apps act on a user's behalf without exposing credentials.
Observability — Ability to understand a system's internal state from its external outputs: logs, metrics, and traces.
ORM — Object-Relational Mapper; library mapping database rows to in-code objects (SQLAlchemy, Hibernate, Prisma).
P
Pagination — Splitting large result sets into pages to avoid returning millions of rows in a single response.
Partitioning — Dividing data across nodes (horizontal) or by type (vertical) to distribute storage and query load.
Polling — Repeatedly asking a source at intervals for new data. Contrasts with push-based webhooks.
Polyglot Persistence — Using different databases for different parts of a system, each chosen for its data model strengths.
Protocol Buffers — Google's language-neutral binary serialization format; faster and smaller than JSON.
Pub/Sub — Publish/Subscribe messaging where publishers emit events to topics and subscribers receive them asynchronously.
Q
Queue — A FIFO data structure or messaging system used for task buffering, async work, and rate smoothing.
R
Race Condition — A bug where behavior depends on the uncontrolled relative timing of two or more concurrent operations.
Rate Limiting — Capping requests per client per time window to protect a service from abuse or overload.
Read Replica — A DB copy that receives writes from the primary and serves reads, increasing read throughput.
Referential Integrity — Constraint ensuring every foreign key value points to an existing primary key row.
Replication — Maintaining synchronized copies of data across multiple nodes for fault tolerance and read scaling.
REST — Representational State Transfer; stateless HTTP architecture using verbs (GET, POST, PUT, DELETE) on noun resources.
Retry Storm — Many clients retrying simultaneously after a failure, amplifying load. Mitigated by jitter and exponential backoff.
S
Saga Pattern — Managing distributed transactions as a sequence of steps, each with a compensating rollback action.
Semantic Versioning — Version scheme MAJOR.MINOR.PATCH: breaking change bumps major, new feature minor, bug fix patch.
Serialization — Converting in-memory data to a storable or transmittable byte format (JSON, Avro, Protobuf).
Service Mesh — Infrastructure layer (Istio, Linkerd) handling service-to-service auth, retries, load balancing, and observability.
Sharding — Horizontally partitioning data across multiple DB instances, each holding a distinct subset.
SOLID — Five OOP design principles: Single responsibility, Open/closed, Liskov substitution, Interface segregation, Dependency inversion.
SQL Injection — Attack embedding malicious SQL into user input that an application executes unescaped against its database.
Stateless — A server storing no per-session client state; any replica can handle any request given the same inputs.
T
Tail Latency — The slowest fraction of requests (p95, p99, p999). Often more damaging to UX than average latency.
Throughput — Operations processed per unit time (req/s, rows/s). Distinct from latency; both can independently be poor.
Time-to-Live — A value specifying when a record (cache entry, DNS record, session token) expires and should be discarded.
Tokenization — Breaking text into smaller units (tokens) for LLM processing; roughly 1 token is ~0.75 English words.
Transaction — A unit of DB work satisfying ACID; either all operations commit or all roll back together.
Two-Phase Commit — Distributed protocol coordinating all participants to either commit or abort a transaction atomically.
U
UUID — Universally Unique Identifier; 128-bit value generated to be globally unique without a central coordinator.
V
Vector Database — Database optimized for storing and similarity-searching high-dimensional embedding vectors at scale.
Vertical Scaling — Upgrading a single machine with more CPU, RAM, or disk. Simpler than horizontal scaling but has a hard ceiling.
Virtual Machine — Software emulation of a full computer system; stronger isolation than containers but heavier to run.
W
Webhook — Server-to-server HTTP callback triggered by an event; the receiver acts on data pushed to it, rather than polling.
WebSocket — Persistent full-duplex TCP connection via HTTP upgrade; enables real-time bidirectional browser-server communication.
Write-Through Cache — Writes update cache and database simultaneously; reads are fast and the cache is never stale on write.
X
XSS — Cross-Site Scripting; injecting malicious scripts into a web page viewed by other users to steal data or sessions.
Z
Zero Downtime Deployment — Deploying a new version while keeping the service continuously available via blue-green, canary, or rolling updates.

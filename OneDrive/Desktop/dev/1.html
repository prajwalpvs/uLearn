from collections import OrderedDict

class LRUCache:
    def __init__(self, cap):
        self.cap = cap
        self.cache = OrderedDict()

    def get(self, key):
        if key not in self.cache:
            return -1
        self.cache.move_to_end(key)
        return self.cache[key]

    def put(self, key, val):
        if key in self.cache:
            self.cache.move_to_end(key)
        self.cache[key] = val

Core Protocols
  • TCP — reliable, ordered, connection-oriented (3-way handshake). HTTP, SSH, databases.
  • UDP — fast, connectionless, no error-check. Streaming, gaming, DNS.
  • HTTP/HTTPS — request-response over TCP. HTTPS adds TLS encryption.

TCP/IP vs OSI
  • TCP/IP (4 layers): Network Access, Internet, Transport, Application — the model the internet actually uses.
  • OSI (7 layers): Physical → Data Link → Network → Transport → Session → Presentation → Application. Used for reasoning and troubleshooting.
TLS Handshake (1.3)
  • Client sends supported cipher suites
  • Server replies with certificate + chosen cipher
  • Key exchange → shared symmetric session key derived. All subsequent traffic encrypted.
  • TLS 1.3 = 1 RTT handshake (TLS 1.2 = 2 RTT — avoid if possible)
When to use TCP vs UDP
  • TCP: anything where dropped packets are unacceptable — web pages, file downloads, API calls, databases.
  • UDP: real-time where stale data is worse than missing — video calls, gaming, live streaming, DNS queries.

Failure Modes
  • Transient errors — temporary (network blip, timeout). Safe to retry with backoff.
  • Non-transient errors — persistent (hardware failure, bad data). Retrying wastes resources.
  • Cascading failures — one failed service causes downstream services to fail in a chain.
► show full design
Retry Strategies
  • Simple retry: for rare transient errors. Cap at 3 attempts.
  • Exponential backoff: 1s → 2s → 4s → 8s. Avoids thundering herd on recovery.
  • Jitter: add random offset to backoff (e.g. ±20%) so retrying clients don't all fire at the same instant.
Circuit Breaker
  • Closed (normal): requests pass through.
  • Open (failing): immediately reject requests — fail fast, don't wait for timeout.
  • Half-open (recovery): let one test request through; if it succeeds, close again.
  • Prevents retry storms and cascading failures.
Saga Pattern
  • Manages distributed transactions across microservices without 2PC.
  • Each step publishes an event or triggers next service. On failure, compensating transactions roll back prior steps.
  • Choreography: services react to events (decentralized, harder to trace).
  • Orchestration: central coordinator drives the saga (visible, easier to debug).
Concept
Availability
Rate Limiting
  • Cap operations per user/IP within a time window. Return 429 when exceeded.
  • Token bucket: N tokens added per second. Allows bursts up to bucket size.
  • Sliding window counter: hybrid — accurate + memory-efficient. Used by Twitter/Redis.
► show full design
Rate Limiting Algorithms
  • Token bucket: tokens refill at rate R; each request consumes 1. Allows short bursts.
  • Leaky bucket: requests drain at constant rate; overflow = rejected. Smooth output.
  • Fixed window: count resets every minute. Simple but allows 2× burst at window edge.
  • Sliding window log: track timestamps of each request. Accurate but memory-heavy.
Queue-Based Load Leveling
  • Introduce a queue between producers and consumers to absorb traffic spikes.
  • Service processes at its own pace; backpressure signals producers to slow down.
  • Increases latency but prevents service overload. Good when out-of-order processing is acceptable.
Gateway Aggregation
  • Gateway fans out to multiple backend services and stitches responses into one reply.
  • Reduces chatty N round-trips (common in mobile) to a single call.
  • Gateway needs circuit breakers and timeouts per sub-call to prevent one slow service from blocking the response.
Concept
Load Balancing
Strategies
  • Round Robin — cycle through servers in order. Simple, even distribution.
  • Least Connections — route to server with fewest active connections. Better for variable request durations.
  • Consistent Hashing — hash(client IP or session) maps to a server. Sticky sessions, CDN routing.
  • Weighted Round Robin — more powerful servers get proportionally more traffic.
► show full design
Layer 4 vs Layer 7
  • L4 (transport): routes by IP + TCP/UDP port. Faster, no content inspection. Cannot route by URL or headers.
  • L7 (application): routes by HTTP URL, headers, cookies. Slower but supports content-based routing, A/B testing, gzip, SSL termination.
Scalability
  • At very high scale, the load balancer itself becomes a bottleneck — run multiple LBs behind Anycast DNS.
  • Horizontal scaling of app servers is easy with stateless services. Stateful sessions need externalised state (Redis) or sticky routing.
Health Checks
  • LB probes backends at /health. Failed servers removed from rotation automatically.
  • Active health check: LB polls. Passive health check: LB watches error rates on live traffic.
  • Combine both for fastest failure detection.
Concept
SQL vs. NoSQL
When to Use Each
  • SQL: complex queries, joins, ACID transactions, structured schema. PostgreSQL, MySQL.
  • NoSQL: flexible schema, horizontal scale, eventual consistency OK. Mongo, Cassandra, DynamoDB.
  • Both: polyglot persistence — SQL for transactions, Redis for cache, Elasticsearch for search.
► show full design
SQL Strengths
  • Relational model — JOINs, foreign keys, referential integrity enforced by DB
  • ACID guarantees — safe for financial transactions, inventory updates
  • Rich query language — aggregations, window functions, subqueries
  • Scaling: primarily vertical. Read replicas for scale-out reads. Sharding adds complexity.
NoSQL Types
  • Document (MongoDB): JSON-like docs, flexible schema. Good for CMS, user profiles.
  • Key-Value (Redis, DynamoDB): O(1) lookup. Sessions, caching, counters.
  • Column-Family (Cassandra): wide rows, write-optimized, massive scale time-series data.
  • Graph (Neo4j): nodes + edges, O(1) per hop. Social graphs, fraud detection.
Decision Scenarios
  • PayPal loan service: SQL — needs ACID, joins between loan, user, transactions.
  • Shopper activity tracking: NoSQL — high write volume, slight staleness OK.
  • Caching customer metrics: Redis — fast, non-relational, TTL-based eviction.
Concept
Database Sharding
Sharding Strategies
  • Hash-based: hash(shard_key) % N. Even distribution, no hotspots. Cross-shard range queries are hard.
  • Range-based: A–M on shard 1, N–Z on shard 2. Simple but uneven if data isn't uniform.
  • Geo-based: data partitioned by user geography. Low latency, but uneven density.
► show full design
Advantages
  • Horizontal scalability — add more servers as data grows
  • Smaller data sets per shard → faster queries, smaller indexes
  • Failure isolation — one shard failing doesn't affect others
Challenges
  • Hotspots: bad shard key choice → one shard gets most traffic. Use consistent hashing or virtual nodes.
  • Cross-shard joins: expensive or impossible. Application must aggregate results.
  • Schema changes: must be rolled out to every shard independently.
  • Rebalancing: adding a shard requires data migration. Consistent hashing minimizes this.
Manual vs Automatic
  • Automatic: CockroachDB, Spanner, Vitess — DB manages shard assignment and rebalancing transparently.
  • Manual: application-level routing. More control but high operational overhead.
Concept
Database Replication
Strategies
  • Leader-Follower: all writes to leader; followers serve reads. Most common.
  • Synchronous replication: leader waits for follower ack. Consistent but slow writes.
  • Asynchronous replication: leader doesn't wait. Fast writes; followers may lag.
  • Multi-Leader / Leaderless (Dynamo): any node can accept writes; conflict resolution required.
► show full design
When to Replicate
  • High read volume: route reads to followers to offload leader.
  • High availability: automatic failover if leader fails (Raft/Paxos elects new leader).
  • Global scale: multi-region replicas reduce latency for geographically distributed users.
Leader Failover
  • Detect failure via heartbeat timeout.
  • Consensus algorithm (Raft/Paxos) or manual promotion elects new leader from followers.
  • Risk: asynchronous follower may not have latest writes → potential data loss window (RPO > 0).
  • Semi-synchronous: at least one follower must ack before leader commits. Balances speed and durability.
Replication Lag
  • Followers may be seconds behind leader — reads from follower can return stale data.
  • Read-your-writes consistency: route reads to leader after a write, or wait for replica to catch up.
  • Monitor lag with SHOW SLAVE STATUS (MySQL) or replication slot lag (Postgres).
Concept
Consistent Hashing
The Problem It Solves
  • Traditional hashing: key → server using hash(key) % N. Adding/removing a server remaps every key — O(n) cache misses.
  • Consistent hashing maps keys AND servers onto a ring. Adding a server only remaps keys between it and its predecessor.
  • On average, only k/n keys remapped when a server is added (k = keys, n = servers).
► show full design
How It Works
  • Hash servers to points on a circular ring (0 to 2³²−1).
  • Hash each key to the same ring. Key is handled by the first server clockwise from it.
  • Adding server: only keys between new server and its predecessor need remapping.
  • Removing server: only keys on that server move to the next clockwise server.
Virtual Nodes
  • A single server maps to many positions (virtual nodes) on the ring.
  • Distributes load more evenly — avoids one server getting a large arc of the ring.
  • More powerful servers get more virtual nodes proportionally.
  • Used by Cassandra, DynamoDB, Memcached.
Use Cases
  • Distributed caches — route requests to correct cache node without remapping on node change.
  • Database partitioning — add shards without reshuffling all data.
  • Load balancers — sticky routing that survives server pool changes.
Concept
CAP Theorem
The Trade-off
  • In a distributed system, you can guarantee at most 2 of 3: Consistency, Availability, Partition Tolerance.
  • In practice, partition tolerance is mandatory — networks always partition eventually. The real choice is C vs A when a partition occurs.
► show full design
Definitions
  • Consistency: every read returns the most recent write or an error.
  • Availability: every request gets a non-error response (may return stale data).
  • Partition Tolerance: system continues operating when network messages are dropped or delayed between nodes.
CP vs AP Systems
  • CP (Consistency + Partition Tolerance): nodes refuse to respond rather than return inconsistent data. Used by: HBase, ZooKeeper, etcd, traditional SQL with strong consistency.
  • AP (Availability + Partition Tolerance): nodes return stale data but keep serving. Used by: Cassandra, DynamoDB, CouchDB — eventual consistency.
PACELC Extension
  • Even without partitions, distributed systems trade Latency vs Consistency.
  • DynamoDB: PA/EL — available under partition, low latency otherwise.
  • Google Spanner: PC/EC — consistent under partition, consistent otherwise (accepts higher latency).
Practical Guidance
  • Financial transactions → CP. Stale money data is unacceptable.
  • Social likes, shopping carts, DNS → AP. Slightly stale is fine; downtime is not.
Concept
Asynchronous Processing
Why It Matters
  • Synchronous: each task blocks until complete. Simple but wastes threads waiting on I/O.
  • Async: tasks operate independently. More throughput, better user experience, resilient to spikes.
  • Key benefit: decouple task submission from execution — smooths bursty load.
► show full design
Processing Patterns
  • Batch processing: accumulate data, process periodically in bulk. High throughput, high latency. MapReduce, Spark.
  • Stream processing: process records in real-time as they arrive. Sub-second latency. Kafka Streams, Flink.
  • Lambda architecture: batch layer (accurate) + speed layer (real-time) merged at query time.
Message Queues
  • Producers publish to queue; consumers pull at their own pace.
  • Decouples services: producer doesn't wait for consumer to finish.
  • Absorbs traffic spikes — queue buffers excess load during peaks.
  • Enables retries: failed jobs re-enqueued; dead letter queue for unprocessable messages.
  • Examples: Kafka (event streaming), RabbitMQ (task queues), SQS (managed), Celery (Python).
Pub/Sub
  • Publishers emit events to a topic without knowing subscribers.
  • Multiple subscribers each receive every message independently.
  • Enables fan-out: one payment event notifies fraud service, email service, analytics simultaneously.
Concept
Caching
Cache Types
  • In-memory (local): process-level cache. Fastest. Per-server, no sharing.
  • Distributed: Redis, Memcached. Shared across servers. Consistent view.
  • CDN: caches static assets at edge nodes close to users geographically.
► show full design
Cache-Aside (Most Common)
  • App checks cache. On miss: fetch from DB, populate cache, return.
  • Cache never gets inconsistent writes, but can serve stale data between TTL expiry and refresh.
Write Strategies
  • Write-through: write to cache + DB simultaneously. Consistent but adds write latency.
  • Write-behind (write-back): write to cache first, DB asynchronously. Fast writes, risk of data loss on crash.
  • Refresh-ahead: proactively refresh cache before TTL expires for hot keys.
Eviction Policies
  • LRU (Least Recently Used): evict item not accessed most recently. Default in Redis.
  • LFU (Least Frequently Used): evict item accessed fewest times. Better for frequency-skewed workloads.
  • TTL: expire entries after a set time regardless of access pattern. Prevents stale data indefinitely.
Cache Stampede Prevention
  • Lock: only one request rebuilds; others wait or serve stale.
  • Jitter: randomize TTL so keys don't expire simultaneously.
  • Probabilistic refresh: start refreshing before expiry using a random dice roll.
Concept
Encryption
Two Main Types
  • Symmetric: same key encrypts and decrypts. Fast. Key must be shared securely. AES-256.
  • Asymmetric: public key encrypts, private key decrypts. Slower but solves key distribution. RSA, ECC.
  • In practice: TLS uses asymmetric to exchange a symmetric key, then uses symmetric for bulk data.
► show full design
Data in Transit
  • TLS (HTTPS) encrypts all HTTP traffic. Certificates verified against CA chain.
  • mTLS (mutual TLS): both client and server authenticate with certificates. Used in service meshes.
  • Perfect Forward Secrecy: ephemeral session keys so past traffic can't be decrypted if private key is later compromised.
Data at Rest
  • AES-256 for encrypting files and database columns.
  • Cloud: server-side encryption (SSE-S3, SSE-KMS) encrypts objects automatically.
  • Customer-managed keys (CMK): you control the key; cloud provider never sees plaintext.
Password Storage
  • Never store plaintext or reversible encryption.
  • Use bcrypt, scrypt, or Argon2 — slow by design to resist brute force.
  • Salt: random value appended to password before hashing — defeats rainbow tables.
  • Work factor: increase cost parameter as hardware speeds up.
Concept
Auth & Authorization
Authentication vs Authorization
  • Authentication (AuthN): who are you? Verifies identity (password, biometric, token).
  • Authorization (AuthZ): what can you do? Checks permissions after identity is confirmed.
  • JWT: signed self-contained token. Server validates signature — no DB lookup per request.
► show full design
Authentication Methods
  • 1FA: username + password only. Weak alone.
  • MFA: password + TOTP (Google Authenticator), SMS, or hardware key (FIDO2). Dramatically reduces account takeover.
  • SSO: one login for many apps. SAML 2.0 (enterprise), OIDC (modern web).
Authorization Models
  • RBAC (Role-Based): users assigned roles (admin, editor, viewer). Roles have permissions. Simplest to manage.
  • ABAC (Attribute-Based): access determined by attributes (user.department == 'finance', resource.classification == 'confidential'). More expressive, harder to audit.
  • ACL: per-resource permission lists. Fine-grained but doesn't scale well.
OAuth 2.0 Flow
  • Authorization Code + PKCE: browser → auth server → authorization code → access token + refresh token. Correct flow for SPAs and mobile apps.
  • Client Credentials: machine-to-machine. App uses its own credentials, no user involved.
  • Access tokens are short-lived (15min). Refresh tokens are long-lived and stored securely.
  • Never store tokens in localStorage — use httpOnly cookies to prevent XSS theft.

Cloud Architecture
Why Cloud
  • Affordability: pay-as-you-go. No upfront hardware capex.
  • Scalability: add capacity in minutes. Scale down when idle — pay only for what you use.
  • Managed services: providers handle OS patching, DB failover, certificate renewal.
  • Global reach: deploy to 30+ regions worldwide in hours.
► show full design
Core Service Categories

  • Storage: object (S3/GCS), block (EBS), file (EFS/Filestore)

  • Re-platform: small changes to use managed services (RDS instead of self-managed MySQL).

Concept
CDNs
What CDNs Do
  • Globally distributed edge servers that cache static content (images, JS, CSS, video) close to end users.
  • Reduces origin server load by 90%+ for media-heavy apps. Reduces latency by serving from the nearest PoP (Point of Presence).
  • Also provides DDoS protection, TLS termination, and WAF at the edge.
► show full design
Push vs Pull CDNs
  • Push: content is manually uploaded to CDN by engineers. Always current, but requires explicit update on every change. Good for large, infrequently-changed assets (video, game patches).
  • Pull (most common): CDN fetches from origin on first cache miss, then serves from cache until TTL expires. Low maintenance; brief stale window on deploy. Used by Cloudflare, CloudFront.
Cache Control
  • Cache-Control: max-age=86400 — browser and CDN cache for 24h.
  • Cache-Control: s-maxage — CDN-specific TTL (different from browser cache).
  • Surrogate-Control / CDN-Cache-Control — vendor-specific headers for fine-grained edge control.
  • Content-addressed URLs (hashed filenames like main.a1b2c3.js) enable indefinite caching — bust cache by deploying new filename.
When Not to Use CDNs
  • Highly dynamic or personalised content that must be fresh per user (financial data, dashboards).
  • Small, localised user base — CDN latency savings are negligible if all users are in one city.
  • Sensitive content that must not be cached on third-party servers.
Popular CDNs
  • Cloudflare CDN, AWS CloudFront, GCP Cloud CDN, Azure CDN, Fastly, Akamai

Distributed Cache
Requirements
  • Low-latency key-value store (sub-millisecond reads)
  • Horizontal scaling to terabytes of data
  • Eviction policies, TTL support
  • High availability — tolerate node failures
► show full design
Data Partitioning
  • Consistent hashing: keys mapped to a virtual ring; each node owns a range — add/remove nodes with minimal remapping (≈ k/n keys moved)
  • Virtual nodes (vnodes) per physical node for more even distribution
Replication
  • Each key replicated to N successor nodes on the ring (typically N=3)
  • Quorum reads/writes (R + W > N) for configurable consistency
  • Read repair and anti-entropy processes fix diverged replicas
Eviction
  • LRU: evict least recently used — good for temporal locality
  • LFU: evict least frequently used — better for skewed access patterns
  • TTL-based expiry via lazy deletion + periodic sweep
Failure Handling
  • Client retries with exponential backoff on connection failure
  • Gossip protocol for node membership and failure detection
  • Hinted handoff: writes for a failed node stored temporarily and replayed on recovery
Thundering Herd
  • Cache miss storm on cold start — mitigated with request coalescing (only one request fills the cache, others wait)
  • Jitter on TTL expiry to prevent synchronized eviction

TCP vs UDP

TCP — Transmission Control Protocol

  • Connection-oriented (SYN → SYN-ACK → ACK handshake)
  • Guaranteed delivery, in-order, error-checked
  • Flow control + congestion control built in
  • Higher overhead — use when reliability matters
  • Use: HTTP, SSH, databases, email (SMTP)

UDP — User Datagram Protocol

  • Connectionless — fire and forget, no handshake
  • No delivery guarantee, no ordering
  • Very low overhead and latency
  • App handles packet loss if needed
  • Use: DNS, video streaming, gaming, VoIP, HTTP/3 (QUIC)

  • Record types: A (→ IPv4), AAAA (→ IPv6), CNAME (alias → canonical name), MX (mail server), TXT (SPF/DKIM)
  • DoH (DNS over HTTPS) and DoT (DNS over TLS) encrypt DNS queries to prevent eavesdropping
HTTP Evolution
  • HTTP/1.1 — persistent connections but HOL blocking; effectively 1 in-flight request per connection despite pipelining
  • HTTP/2 — binary framing; multiplexing (many streams per TCP conn); header compression (HPACK); server push
  • HTTP/3 — runs on QUIC (UDP-based); eliminates TCP HOL blocking; 0-RTT session resumption; built-in TLS 1.3; better on lossy networks
TLS 1.3 Handshake
  • ClientHello — supported ciphers, DH key share, random nonce
  • ServerHello — chosen cipher, server key share, certificate chain
  • Client verifies cert against trusted CA roots in browser/OS store
  • Session keys derived via Diffie-Hellman — secret never transmitted
  • TLS 1.3: 1 RTT total (vs 2 for TLS 1.2); 0-RTT for session resumption
  • PFS: ephemeral keys mean past traffic can't be decrypted if private key leaks later
WebSockets vs SSE vs Polling
  • WebSocket — full-duplex, persistent; upgrades from HTTP via Upgrade: websocket; use for chat, games, live collab
  • SSE (Server-Sent Events) — server-to-client only; uses plain HTTP; simpler; reconnects automatically; use for dashboards, notifications
  • Long polling — client holds request open; server replies when data ready; fallback for restricted environments
  • Scale WebSockets horizontally with sticky sessions or shared Redis pub/sub
Load Balancers: L4 vs L7
  • L4 (Transport) — routes by IP + TCP/UDP port; no content inspection; very fast; e.g. AWS NLB, HAProxy TCP
  • L7 (Application) — routes by URL path, headers, cookies; SSL termination; A/B routing; e.g. AWS ALB, nginx, Envoy
  • Algorithms: Round Robin, Least Connections, IP Hash (sticky), Weighted RR
  • Health checks: periodic probes remove unhealthy backends automatically
  • CDN (Cloudflare, CloudFront) = globally distributed L7 cache + anycast edge routing
Authentication vs Authorization

Authentication (AuthN) — Who are you?

  • Prove identity: password, biometric, hardware key, OTP
  • Result: session cookie, JWT, signed token
  • Protocols: OAuth 2.0, OIDC, SAML, LDAP
  • Factors: something you know / have / are → MFA

Authorization (AuthZ) — What can you do?

  • Checks permissions after identity confirmed
  • RBAC — roles (admin, editor, viewer)
  • ABAC — attributes (dept=eng, clearance=3)
  • Principle of Least Privilege — grant minimum needed
  • Tools: OPA (Open Policy Agent), Casbin, AWS IAM
JWT — JSON Web Tokens
Structure: header.payload.signature (all base64url encoded)
// Header — algorithm + token type
{ "alg": "RS256", "typ": "JWT" }

// Payload — claims (public + private)
{ "sub": "user_123", "role": "admin", "exp": 1735689600 }

// Signature — server verifies on every request
RSASHA256(base64url(header) + "." + base64url(payload), privateKey)
  • Stateless — server validates signature without DB lookup (ideal for microservices)
  • Prefer RS256 (asymmetric) over HS256 (shared secret) for distributed systems
  • Short expiry (15 min access token) + long-lived refresh token + rotation = revocability
  • Store in HttpOnly cookie, NOT localStorage — prevents XSS token theft
  • Payload is base64-encoded, not encrypted — never put secrets in JWT claims
OAuth 2.0 Grant Types
Authorization Code + PKCE (web / mobile)
  • User authenticates at AuthServer → redirected back with a one-time code
  • Client exchanges code + code_verifier (PKCE) for access + refresh tokens
  • PKCE prevents code interception attacks — safe for public clients (SPA, mobile)
  • Never expose client_secret in frontend JS — use PKCE instead
Client Credentials (Machine-to-Machine)
  • Service sends client_id + client_secret directly to AuthServer for token
  • No user involved — for backend-to-backend API calls
  • Rotate secrets via secrets manager; never hardcode in source code
  • Scope access narrowly — one client per service per environment
Password Hashing
Use bcrypt / Argon2id / scrypt — never raw MD5, SHA1, or SHA256
import bcrypt

# Hash on registration (cost=12 ≈ 300ms — slow is the point)
hashed = bcrypt.hashpw(password.encode(), bcrypt.gensalt(rounds=12))

# Verify on login — uses constant-time comparison internally
is_valid = bcrypt.checkpw(candidate.encode(), hashed)
  • bcrypt automatically salts each hash — no separate salt management needed
  • Argon2id is the 2023 OWASP recommendation — memory-hard, resistant to GPU cracking
  • MD5 / SHA-256 are fast checksums for data integrity, not designed for passwords
  • Increase cost factor over time as hardware improves; re-hash on next login
OWASP Top 10 Essentials
SQL Injection
# Vulnerable — attacker passes id="1 OR 1=1"
query = f"SELECT * FROM users WHERE id={user_id}"

# Safe — parameterized query
cursor.execute("SELECT * FROM users WHERE id = %s", (user_id,))

Use ORMs or parameterized queries everywhere. Grant DB accounts only the permissions they need. Validate and whitelist all user input at the boundary.

XSS — Cross-Site Scripting
// Vulnerable — injects attacker JS into DOM
element.innerHTML = userInput;

// Safe — treats as plain text
element.textContent = userInput;
// Rich text: sanitize with DOMPurify first

Escape all user-controlled output in templates. Set Content-Security-Policy header. Use HttpOnly cookies so JS cannot read auth tokens even if XSS occurs.

CSRF — Cross-Site Request Forgery
  • Attacker's page tricks logged-in user's browser into sending a forged request to your API
  • Defense 1: CSRF token — random, per-session, validated server-side on every mutating request
  • Defense 2: SameSite=Strict or SameSite=Lax cookie attribute blocks cross-origin sends
  • Defense 3: Verify Origin / Referer headers on state-changing endpoints
Cookie Security Flags + CORS
  • HttpOnly — JS cannot read cookie; blocks XSS token theft
  • Secure — only sent over HTTPS; never over HTTP
  • SameSite=Strict — never sent cross-site (may break OAuth redirects)
  • SameSite=Lax — sent on top-level navigation only (safe default)
  • CORS: Access-Control-Allow-Origin restricts which origins can call your API; never use * with credentials

How Indexes Work
B-Tree Index
  • Balanced tree where leaf nodes hold data pointers; internal nodes hold separator keys
  • Height ≈ logf(n) where f = branching factor (100s–1000s) — typically 3–4 levels for millions of rows
  • Supports: equality, range (>, BETWEEN), prefix (LIKE 'foo%'), ORDER BY
  • Write overhead: insert/update must rebalance tree — splits and merges pages
  • Used by: PostgreSQL, MySQL InnoDB (default index type)
Hash Index
  • Hash of key → pointer to row. O(1) exact lookups.
  • Does NOT support range queries or sorting
  • PostgreSQL builds in-memory hash indexes for hash joins; on-disk hash indexes exist but rarely used
  • Redis, Memcached use hash tables as their primary structure
LSM-Tree (Log-Structured Merge)
  • Writes go to an in-memory buffer (MemTable) + WAL; periodically flushed to immutable SSTable files on disk
  • Background compaction merges SSTables and removes tombstones
  • Reads may check multiple SSTables — Bloom filters reduce unnecessary disk reads
  • Write-optimized: sequential I/O only. Used by: RocksDB, Cassandra, LevelDB, HBase
GIN / Full-Text Index
  • Generalized Inverted Index — maps each token to list of rows containing it
  • Used for full-text search (tsvector), JSONB key lookup, array containment in Postgres
  • Slower to update than B-tree; use fastupdate with pending list for write performance

Query Planning & EXPLAIN
How the Planner Works
  • Parses SQL → logical plan → enumerate physical plans → cost each → pick cheapest
  • Cost = (seq_page_cost × pages) + (cpu_tuple_cost × rows) — tunable parameters
  • Statistics drive estimates: pg_stats stores column histograms, most-common values, null fraction
  • Run ANALYZE after bulk loads to refresh statistics
Reading EXPLAIN ANALYZE
  • Seq Scan — full table scan; fine for small tables or when returning most rows
  • Index Scan — uses index, fetches heap for each row; good for selective queries
  • Index Only Scan — no heap fetch; fastest possible read path
  • Bitmap Heap Scan — multiple index conditions combined; fetches pages in physical order
  • Hash Join — builds hash table on smaller relation; best for large unsorted joins
  • Nested Loop — for each row in outer, probe inner; great when inner is small + indexed
  • Merge Join — both sides sorted; efficient for large sorted inputs
  • Look at actual rows vs estimated rows — large divergence = stale stats or missing index
Common Slowdown Causes
  • Missing index on filter/join column — add it, then re-explain
  • Stale statistics — run ANALYZE table
  • N+1 queries — batch with IN or JOIN instead of querying per-row in app code
  • SELECT * — fetches all columns including large text/blob fields; select only what you need
  • Unbounded pagination (OFFSET 10000) — use keyset/cursor pagination instead
  • Lock contention — long-running transactions block writes; keep transactions short
  • Connection exhaustion — use a connection pool (PgBouncer); don't open one connection per request

Normalization vs Denormalization

Normalized (3NF)

  • No redundant data — each fact in one place
  • Easy updates, no anomalies
  • Requires JOINs to reconstruct data — slower reads
  • Use for OLTP where writes are frequent

Denormalized

  • Duplicate data for faster reads — fewer JOINs
  • Updates must touch multiple places (consistency risk)
  • Use for OLAP / reporting, or read-heavy production APIs
  • Common pattern: materialize aggregates or embed nested objects in Postgres JSONB
APIs — How Computers Talk to Each Other

Just like people communicate through shared conventions, computers communicate through APIs (Application Programming Interfaces). An API is a defined contract specifying how two systems can talk. There are several styles — each with different tradeoffs around flexibility, efficiency, and simplicity.

REST API
  • What it is: Stateless, client-server architecture using standard HTTP methods on named resource URLs
  • Analogy: Ordering from a fixed restaurant menu — you pick from predefined items; the waiter (server) fetches exactly what you asked for
  • Resources identified by URL: /users/42, /orders/99
  • Each request is self-contained — server holds no session state between calls
  • Response format is typically JSON; easy to debug and inspect in any browser
  • Best for: Public APIs, CRUD apps, mobile backends — anything where simplicity and broad client support matter
GraphQL
  • What it is: A query language for APIs where the client defines the exact shape of the data it needs in a single request
  • Analogy: Ordering a fully custom burger in one go — extra pickles, gluten-free bun, no sauce — the kitchen assembles exactly that. REST would make you order each ingredient separately and assemble it yourself
  • Eliminates over-fetching (getting fields you don't need) and under-fetching (needing multiple round-trips)
  • Single endpoint: POST /graphql — the query body defines what comes back
  • Strongly typed schema — clients know every available field and type at development time
  • Best for: Complex UIs with many data shapes, mobile clients on constrained bandwidth, teams where frontend and backend iterate independently
gRPC
  • What it is: Remote Procedure Call framework using Protocol Buffers (binary) over HTTP/2 for high-efficiency service-to-service communication
  • Analogy: Kitchen staff using shorthand — "B+F table 1" instead of reading out the full order. Internally fast; not human-friendly but extremely efficient
  • Proto files define the schema and auto-generate client/server code in any language
  • Supports streaming: unary (1 req → 1 resp), server-streaming, client-streaming, bidirectional
  • 3–10× smaller payload and significantly faster parse time vs JSON REST
  • Best for: Internal microservice-to-microservice calls, low-latency scenarios, polyglot environments (Go ↔ Java ↔ Python)
SOAP
  • What it is: Strict protocol using XML envelopes for messaging; built-in standards for security (WS-Security), transactions, and reliability
  • Heavy and verbose — every request/response is XML-wrapped with headers, body, and optional fault sections
  • Formally described by WSDL (Web Services Description Language) — machine-readable contract
  • Best for: Enterprise integrations, banking/finance where formal contracts and built-in WS-* standards are required. Rarely chosen for new projects today.

Message Queues

Without a Queue (Synchronous)

  • Producer calls consumer directly and waits for a response
  • If consumer is slow or down, producer is blocked
  • Both services must be available at the same time
  • Traffic spikes overwhelm the consumer immediately
  • Like a homeowner waiting beside his maid until each chore is done before assigning the next — he never makes it to work on time

With a Queue (Asynchronous)

  • Producer drops messages into the queue and moves on immediately
  • Consumer processes at its own pace — services are fully decoupled
  • Queue acts as a buffer — absorbs traffic spikes without overwhelming the consumer
  • Consumer can be offline; messages wait safely in the queue
  • Like writing a task checklist and leaving for work — the maid processes chores independently, nothing is lost
Key Concepts
  • Producer — publishes messages to the queue (the homeowner writing the checklist)
  • Consumer — reads and processes messages from the queue (the maid completing tasks)
  • Message — the unit of work: JSON payload, event, command, or raw bytes
  • Queue / Topic — the buffer that holds messages until consumed
  • Acknowledgement (ACK) — consumer signals the queue that processing succeeded; unACKed messages are redelivered
  • Dead Letter Queue (DLQ) — messages that fail repeatedly are moved here for inspection
Benefits
  • Decoupling: producer and consumer evolve independently — change one without touching the other
  • Traffic smoothing: queue absorbs sudden spikes; consumer processes at a steady rate
  • Reliability: messages persist in the queue even if the consumer crashes — no work is lost
  • Scalability: add more consumers to process faster; each pulls from the shared queue
  • Retry semantics: failed messages re-enter the queue automatically with configurable backoff
Drawbacks
  • Added latency: not suitable for tasks requiring an immediate response (e.g. user login, payment confirmation shown instantly)
  • Overhead for trivial tasks: serializing, queuing, and consuming a message costs more than a direct function call for simple operations
  • Eventual consistency: consumers process asynchronously — the system is eventually consistent, not immediately
  • Operational complexity: queue infra (Kafka brokers, RabbitMQ nodes) must be monitored, scaled, and maintained
  • Message ordering: not guaranteed across partitions in Kafka; requires careful key-based partitioning if order matters
Queue vs Pub/Sub
  • Queue (point-to-point): one message → one consumer. E.g. SQS, RabbitMQ queue. Good for task distribution — each job processed exactly once.
  • Pub/Sub (fan-out): one message → many consumers. E.g. Kafka topic, Google Pub/Sub. Good for event broadcasting — all subscribers get notified.
  • Kafka blends both: a topic is pub/sub across consumer groups, but within a consumer group it behaves like a queue (each partition → one consumer)

A
ACID — Atomicity, Consistency, Isolation, Durability: the four guarantees a relational DB transaction must satisfy. Violations of any one can corrupt data or allow dirty reads.
Abstraction — Hiding implementation complexity behind a clean interface so callers only see what they need. Good abstractions let you swap internals without breaking callers.
Actor Model — A concurrency model where computation is split into independent actors communicating only via asynchronous message passing with no shared mutable state. Each actor processes its mailbox one message at a time, eliminating data races. Used by Erlang/Elixir and the Akka framework.
Adjacency List — Graph representation where each node stores a list of its neighbors. Space-efficient for sparse graphs: O(V+E) vs O(V²) for an adjacency matrix.
Agile — Iterative delivery methodology built around short sprints, continuous feedback, and working software over documentation. Common frameworks: Scrum, Kanban, SAFe.
Amdahl's Law — Theoretical speedup limit: if fraction f of work is parallelizable, max speedup = 1 / (1 - f). 90% parallel work → max 10× speedup regardless of core count.
Amortized Analysis — Average cost per operation over a sequence rather than per-operation worst case. Dynamic array append is O(n) occasionally but O(1) amortized across all appends.
API — Application Programming Interface; a defined contract specifying how two systems communicate. REST, GraphQL, and gRPC are common API styles for web services.
Asynchronous — Operations that return immediately; the caller handles the result later via callback, promise, or event. Enables concurrency without blocking threads.
Atomicity — A transaction either fully completes or fully rolls back; no partial intermediate state is ever left visible. The "A" in ACID.
B
Backpressure — A flow-control signal where a downstream consumer tells an upstream producer to slow down. Prevents memory exhaustion when consumers can't keep up with producers.
Batch Processing — Processing accumulated data in large chunks on a schedule rather than record-by-record in real time. Hadoop and Spark are canonical batch frameworks; cheaper but higher latency than streaming.
BFS / DFS — Breadth-First Search explores all neighbors at depth d before d+1 (uses a queue; finds shortest path in unweighted graphs). Depth-First Search goes deep first (uses a stack or recursion; useful for cycle detection, topological sort).
Big-O Notation — Describes the worst-case growth rate of an algorithm's time or space as input size n grows. Drop constants and lower-order terms: 3n² + n → O(n²).
Binary Search — O(log n) search on a sorted array by repeatedly halving the search range. Precondition: input must be sorted. Template: lo, hi, mid = lo + (hi - lo) // 2.
Bloom Filter — A probabilistic data structure that answers "possibly in set" or "definitely not in set" using minimal memory. False positives are possible; false negatives are not. Used by Cassandra, Redis, and CDNs to avoid unnecessary lookups.
Blue-Green Deployment — Two identical production environments; deploy to the idle one, then switch traffic instantly for zero downtime. Rollback = flip traffic back to old environment.
B-Tree — Self-balancing tree where each node can have many children (branching factor 100s–1000s). Height ≈ log_f(n) — typically 3–4 levels for millions of rows. The default index structure in PostgreSQL, MySQL, and most RDBMS.
Branching Strategy — A convention for creating and merging code branches. Git Flow uses long-lived feature/release branches; trunk-based development keeps one main branch with short-lived feature branches.
C
CAP Theorem — In a distributed system you can guarantee at most two of: Consistency, Availability, Partition Tolerance. In practice all systems must tolerate partitions, so the real tradeoff is C vs A.
Cache Invalidation — Removing or updating stale cached data when the source of truth changes. Notoriously hard — strategies include TTL-based expiry, write-through, and event-driven invalidation.
Canary Release — Rolling a new version out to a small fraction of users first to catch problems before full deployment. Traffic is gradually shifted (1% → 10% → 100%) while monitoring error rates.
CDN — Content Delivery Network; globally distributed edge servers that cache static assets close to end users. Reduces origin load by 90%+ for media-heavy applications. Examples: Cloudflare, Akamai, CloudFront.
Circuit Breaker — Stops forwarding requests to a failing dependency, failing fast and returning an error immediately. After a cooldown it enters "half-open" state to test recovery. Prevents cascading failures.
Closure — A function bundled with the variables from its enclosing scope, captured at definition time. Even after the outer function returns, the closure can still access those variables. A common source of memory leaks when closures inadvertently hold large objects alive.
Cold Start — The delay when an idle serverless function or container must initialize before handling its first request. Mitigated by keeping instances warm via scheduled pings or provisioned concurrency.
Compaction — Background process in LSM-tree databases (RocksDB, Cassandra) that merges SSTables, removes deleted tombstones, and reclaims disk space. Trades I/O bandwidth for read performance.
Connection Pool — A pre-created set of database connections reused across requests. Avoids the ~50ms overhead of opening a new TCP+TLS+auth handshake per request. Tools: PgBouncer, HikariCP, SQLAlchemy pool.
Consistency — Every read sees the most recent committed write, or returns an error. Strong consistency is expensive across nodes; eventual consistency relaxes this for availability.
Consistent Hashing — Hashing technique mapping both keys and nodes to a ring, so adding or removing a node remaps only k/n keys on average. Used in distributed caches (Memcached, DynamoDB) and load balancers.
Container — Lightweight portable runtime package bundling app code and dependencies, isolated from the OS but sharing the host kernel. Faster to start than VMs (milliseconds vs seconds). Docker is the dominant runtime.
CQRS — Command Query Responsibility Segregation; using separate models for reads and writes so each can be optimized and scaled independently. Often paired with event sourcing.
CSRF — Cross-Site Request Forgery; an attack that tricks a user's authenticated browser into sending an unwanted request to another site. Prevented by the SameSite cookie attribute, CSRF tokens (synchronized token pattern), or verifying the Origin/Referer header.
CRUD — Create, Read, Update, Delete: the four fundamental operations on persistent data. Maps to HTTP verbs POST, GET, PUT/PATCH, DELETE in a REST API.
D
Data Lake — A storage repository (typically S3 or GCS) holding raw, unstructured, or semi-structured data at any scale. Schema is applied on read, making it flexible but requiring careful governance to avoid a "data swamp."
Data Warehouse — Structured, query-optimized columnar storage for analytics (Snowflake, BigQuery, Redshift). Schema on write, fast aggregations over billions of rows. Contrasts with data lakes which accept raw, schema-less data.
Dead Letter Queue — A queue where unprocessable messages land after repeated delivery failures. Isolates bad messages so healthy messages aren't blocked. Ops teams inspect and replay or discard DLQ entries.
Deadlock — Two or more processes each waiting for a lock held by the other; none can proceed. Databases detect and kill one transaction. Prevention: always acquire locks in the same order.
Declarative — You describe the desired end-state, not the steps to reach it. SQL, Kubernetes YAML, and Terraform are declarative. Contrasts with imperative code that specifies every step.
Dependency Injection — Passing a component's dependencies from outside rather than having it construct them internally. Improves testability (swap real DB for mock) and follows the Dependency Inversion principle.
DevOps — Culture and practice uniting development and operations to shorten release cycles and improve reliability. Key practices: CI/CD, infrastructure as code, monitoring, blameless post-mortems.
DNS — Domain Name System; translates human-readable hostnames to IP addresses via a distributed hierarchy of resolvers and authoritative nameservers. Record types: A, AAAA, CNAME, MX, TXT.
DRY — Don't Repeat Yourself: every piece of knowledge should have a single authoritative representation in a system. Duplication means two places to update when logic changes — and one will be forgotten.
Dynamic Programming — Optimization technique solving problems by breaking them into overlapping subproblems and caching results to avoid recomputation. Two approaches: top-down memoization (recursive) or bottom-up tabulation (iterative).
E
Embedding — A dense vector representation of text, images, or other data in high-dimensional space. Semantically similar items cluster close together — the foundation of semantic search, RAG, and recommendation systems.
Encapsulation — Bundling data with the methods that operate on it and restricting direct access to internals. Callers interact via a public interface; implementation details can change without breaking them.
ETL — Extract, Transform, Load: the classic pipeline for moving data from source systems into a warehouse. ELT (Extract, Load, Transform) loads raw data first and transforms inside the warehouse — favored by modern cloud warehouses.
Event Loop — The mechanism enabling non-blocking I/O in single-threaded environments like Node.js and the browser. Runs continuously: execute call stack → drain microtask queue (Promises, queueMicrotask) → run next macrotask (setTimeout, I/O callbacks) → repeat. Long synchronous work blocks the entire loop.
Event Sourcing — Storing state as an append-only log of events; current state is reconstructed by replaying them. Enables audit trails, temporal queries, and event-driven projections. Often paired with CQRS.
Eventual Consistency — Replicas will converge to the same value given sufficient time and no new writes. Traded for availability in AP systems (Cassandra, DynamoDB). Acceptable for social likes, not for bank balances.
Exponential Backoff — Retry strategy where the wait time doubles after each failure (e.g. 1s, 2s, 4s, 8s). Add random jitter to prevent all clients retrying simultaneously and creating a retry storm.
F
FaaS — Function-as-a-Service; cloud execution model running individual functions in response to events, billed per invocation with automatic scaling to zero. AWS Lambda, Google Cloud Functions, and Azure Functions are the main platforms. Cold start latency is the primary trade-off versus always-on containers.
Fan-out — Distributing one event or write to many downstream recipients simultaneously. E.g. pushing a new tweet to all followers' feed lists in Redis. High fan-out (celebrities with millions of followers) requires special handling.
Feature Flag — A runtime switch enabling or disabling a feature without a code deployment. Enables trunk-based development, A/B testing, and instant rollback of problematic features. Tools: LaunchDarkly, Unleash.
FIFO — First In, First Out: the ordering discipline of a queue. The first element enqueued is the first dequeued. Implemented with a doubly linked list or circular array for O(1) enqueue and dequeue.
Foreign Key — A column referencing the primary key of another table, enforcing referential integrity. The DB rejects inserts that reference non-existent rows and can cascade deletes to dependent rows.
Functional Programming — A paradigm treating computation as pure function evaluation with no side effects or shared mutable state. Makes code easier to reason about and test. Key concepts: immutability, higher-order functions, currying.
G
Garbage Collection — Automated runtime reclamation of memory held by objects no longer reachable from any reference. Strategies: mark-and-sweep, generational GC (young/old generation), reference counting. GC pauses can cause latency spikes.
Goroutine — A lightweight concurrent function in Go, managed by the Go runtime rather than the OS. Goroutines start with ~2 KB of stack (vs ~1 MB for OS threads) and are multiplexed onto a small pool of OS threads by Go's scheduler. Communicate via channels rather than shared memory.
Gossip Protocol — Decentralized protocol where nodes periodically exchange state with random peers. Information spreads exponentially like a rumor. Used by Cassandra and DynamoDB for cluster membership and failure detection.
Graph Database — Database optimized for highly connected data using nodes, edges, and properties. Traversals that would require many JOINs in SQL are O(1) per hop. Examples: Neo4j, Amazon Neptune. Used for social graphs, fraud detection, knowledge graphs.
GraphQL — A query language for APIs where clients specify exactly the data shape they need in a single request. Eliminates over-fetching (too many fields) and under-fetching (too few — requiring N+1 calls) common with REST.
gRPC — Google's high-performance RPC framework using Protocol Buffers over HTTP/2. Supports unary, server-streaming, client-streaming, and bidirectional-streaming calls. Strongly typed schema via .proto files.
H
Hash Function — Maps arbitrary input to a fixed-size output deterministically; identical inputs always produce identical outputs. A good hash function distributes outputs uniformly to minimize collisions in hash maps.
Head-of-Line Blocking — When one slow or stalled request prevents all subsequent requests behind it from being processed. HTTP/1.1 suffers HOL blocking per connection; HTTP/2 multiplexing fixes it at the HTTP layer; HTTP/3 (QUIC over UDP) fixes it at the transport layer.
Heap (data structure) — A complete binary tree satisfying the heap property: in a min-heap every parent ≤ its children. Supports O(log n) insert and O(1) peek at min/max. Used for priority queues, Dijkstra's algorithm, and top-k problems.
HMAC — Hash-based Message Authentication Code; proves both integrity and authenticity of a message using a shared secret key. Used to sign webhook payloads (receivers verify before trusting), JWT signatures, and AWS Signature v4. Unlike a plain hash, cannot be forged without the secret.
HNSW — Hierarchical Navigable Small World: approximate nearest-neighbor graph index for fast similarity search in high-dimensional vector spaces. Used by Pinecone, Weaviate, and pgvector. Trades small recall loss for orders-of-magnitude speedup over brute force.
Horizontal Scaling — Adding more machines (scale out) to distribute load rather than upgrading one machine (scale up). Requires stateless services or distributed state management. No hard ceiling but adds coordination complexity.
Hot Spot — A shard or partition receiving disproportionately high traffic, creating a localized bottleneck. Common causes: sequential IDs used as shard keys, celebrity accounts. Fixed by random prefixes, virtual nodes, or shard splitting.
HTTP/2 — HTTP revision enabling multiplexing of many requests over one TCP connection, header compression (HPACK), and server push. Eliminates head-of-line blocking at the HTTP layer (though TCP HOL blocking remains; HTTP/3 fixes this with QUIC).
Hysteresis — Intentional lag before reacting to a state change to prevent rapid oscillation. Autoscalers use hysteresis to avoid thrashing — scale up fast, scale down slowly after a cooldown period.
I
Idempotency — An operation that produces the same result whether applied once or many times. Critical for safe retries: charging a payment once even if the request is sent twice. Implemented via idempotency keys.
Immutability — Data that cannot be changed after creation; updates produce new values rather than modifying existing ones. Eliminates a class of concurrency bugs and makes caching safe. Foundation of functional programming and event sourcing.
Indexing — An auxiliary data structure (B-tree, hash, GIN) letting a DB locate rows without scanning the full table. Speeds reads at the cost of write overhead and storage. Every index must be updated on INSERT/UPDATE/DELETE.
Infrastructure as Code — Managing infrastructure via version-controlled declarative config files (Terraform, CloudFormation, Pulumi) instead of manual steps. Enables reproducible environments, code review, and automated provisioning.
Inverted Index — Maps each term to a sorted list of documents containing it (posting list). The core data structure behind search engines and Elasticsearch. Allows AND/OR queries by intersecting or unioning posting lists.
Isolation Level — Database setting controlling how much one transaction is shielded from others. Levels: Read Uncommitted → Read Committed → Repeatable Read → Serializable. Higher isolation = fewer anomalies but more lock contention and lower throughput.
J
JIT Compilation — Just-In-Time compilation; translating bytecode or interpreted code into native machine instructions at runtime rather than before execution. The JVM's HotSpot compiler and V8's TurboFan JIT let managed languages achieve near-native performance on hot code paths.
Job Queue — A queue storing units of background work (jobs) consumed by worker processes asynchronously. Decouples job submission from execution; allows retries, priority, and scheduling. Examples: Celery, Sidekiq, BullMQ.
JSON — JavaScript Object Notation; human-readable key-value text format universally used for REST API payloads and config files. More verbose than binary formats (Protobuf, Avro) but far easier to debug and inspect.
JWT — JSON Web Token; a signed self-contained token in three Base64-encoded parts: header.payload.signature. Widely used for stateless authentication — the server verifies the signature without a DB lookup. Store in httpOnly cookies, not localStorage.
K
Kafka — Distributed event-streaming platform storing messages in ordered, partitioned, replicated logs. Consumers pull at their own pace and messages are retained for days. Used for decoupling services, event sourcing, and stream processing pipelines.
Key-Value Store — Simplest NoSQL model: a map from opaque keys to opaque values. Extremely fast O(1) lookups; no relational query support. Examples: Redis (in-memory), DynamoDB (distributed), etcd (strongly consistent).
Kubernetes — Container orchestration platform automating deployment, scaling, and management of containerized workloads. Core primitives: Pod (smallest deployable unit), Deployment (desired replica count), Service (stable network endpoint), Ingress (HTTP routing).
L
Latency — Time elapsed for a single request to travel end-to-end (client → server → client). Measured in ms. Distinct from throughput — a system can have high throughput but also high latency. Always chase p99, not just the average.
Leader Election — Distributed process by which nodes agree on a single coordinator to avoid split-brain. Implemented via ZooKeeper, etcd, or Raft consensus. The leader handles writes; followers replicate and serve reads.
LIFO — Last In, First Out: the ordering discipline of a stack. The most recently pushed element is popped first. Used in undo history, call stacks, DFS traversal, and expression parsing.
Load Balancer — Distributes incoming requests across multiple backend servers. Algorithms: round-robin, least connections, IP hash (sticky sessions). Layer 4 LBs route by IP/TCP; Layer 7 LBs can route by URL, headers, or cookies.
Long Polling — Client sends a request and the server holds it open until data is available or a timeout occurs, then the client immediately reconnects. More efficient than short polling; simpler than WebSocket. Used by Slack and early Facebook chat for real-time notifications.
LoRA — Low-Rank Adaptation; efficient LLM fine-tuning by inserting small trainable matrices (rank r) into frozen model weight layers. Reduces trainable parameters by 10,000× vs full fine-tuning with comparable quality.
LRU Cache — Least Recently Used eviction policy: removes the item accessed least recently when the cache is full. Implemented in O(1) with a hash map + doubly linked list. Variants: LFU (least frequently used), ARC (adaptive).
LSM-Tree — Log-Structured Merge-tree: write-optimized storage where all writes go to an in-memory MemTable + WAL, then flush to immutable SSTable files on disk. Background compaction merges files. Used by RocksDB, Cassandra, LevelDB.
M
MapReduce — Parallel batch-processing model: Map phase applies a function to each input record emitting key-value pairs; Reduce phase aggregates values by key. Foundation of Hadoop; Spark generalizes it with an in-memory DAG engine.
Marshalling — Converting an in-memory object into a transmittable or storable format (JSON, Protobuf, Avro). The reverse is unmarshalling (or deserialization). Binary formats are faster and smaller; text formats are more debuggable.
Materialized View — A precomputed, persisted query result stored on disk and refreshed periodically or on change. Trades read speed for staleness risk. Useful for complex aggregation queries that are too slow to compute on demand.
Memoization — Caching function results keyed by input so identical calls return the cached value instead of recomputing. Top-down dynamic programming uses memoization. Only works for pure (side-effect-free) functions.
Message Queue — Async communication buffer where producers publish messages and consumers retrieve and process them independently at their own pace. Decouples services, absorbs traffic spikes, and enables retries. Examples: Kafka, RabbitMQ, SQS.
Middleware — Software layer sitting between application components to handle cross-cutting concerns — authentication, logging, rate limiting, request parsing — without mixing them into business logic. In Express.js a middleware is a function receiving (req, res, next); in backend systems it may be an API gateway or service mesh.
Microservices — Architectural style decomposing an application into small, independently deployable services each owning its own data store. Enables independent scaling and deployment but adds network latency, distributed tracing complexity, and operational overhead.
Monolith — A single deployable unit containing all application components. Simpler to develop, test, and debug early on; harder to scale individual components independently as the codebase grows.
Mutex — Mutual-exclusion lock ensuring only one thread accesses a shared resource at a time. A thread acquires the mutex before entering a critical section and releases it after. Misuse causes deadlocks; overuse causes contention.
MVCC — Multi-Version Concurrency Control: each transaction sees a snapshot of committed data at its start time, so readers never block writers and writers never block readers. Used by PostgreSQL, MySQL InnoDB, CockroachDB.
N
N+1 Problem — Fetching N records then issuing one extra query per record, totaling N+1 queries. Causes exponential DB load as N grows. Fix: batch with IN (…) or JOIN, or use a DataLoader-style batching library.
Namespace — An isolated scope that prevents name collisions between resources. In Kubernetes, namespaces partition cluster resources between teams. In programming languages, packages and modules provide namespace isolation. In Linux, namespaces are the kernel primitive underlying container isolation (PID, network, mount namespaces).
Normalization — Organizing relational tables to eliminate redundancy and update anomalies. Normal forms: 1NF (atomic columns), 2NF (no partial dependencies), 3NF (no transitive dependencies). Over-normalization hurts read performance — denormalize strategically.
NoSQL — Non-relational databases trading some SQL guarantees for scale or flexibility. Four families: document (MongoDB), key-value (Redis), column-family (Cassandra), graph (Neo4j). Each optimizes for a different data model and access pattern.
NP-Hard — A class of problems at least as hard as the hardest problems in NP. No known polynomial-time exact solution exists. Examples: Traveling Salesman, 0/1 Knapsack, graph coloring. In practice: use greedy heuristics or approximation algorithms.
O
OAuth 2.0 — Authorization framework letting third-party apps act on a user's behalf without exposing credentials. Core flows: Authorization Code (with PKCE for SPAs), Client Credentials (machine-to-machine). Access tokens are short-lived; refresh tokens long-lived.
Observability — Ability to understand a system's internal state from its external outputs. The three pillars: logs (what happened), metrics (how much / how fast), traces (where time was spent across services).
OIDC — OpenID Connect; identity layer on top of OAuth 2.0 providing authentication in addition to authorization. Returns an ID token (JWT with user claims) alongside the access token. Used by "Sign in with Google/Apple."
ORM — Object-Relational Mapper; library mapping database rows to in-code objects (SQLAlchemy, Hibernate, Prisma). Reduces boilerplate but can generate inefficient queries — always check the SQL your ORM emits in production.
Outbox Pattern — Write a business record and an outbox event atomically in one DB transaction; a separate poller publishes the event to a message broker. Solves the dual-write problem — guarantees the event is published if and only if the DB write commits.
P
PACELC — Extension of CAP: even without partitions, distributed systems must tradeoff Latency vs Consistency. DynamoDB defaults to low latency (EL); Spanner defaults to high consistency (EC).
Pagination — Splitting large result sets into pages. Offset pagination (LIMIT/OFFSET) is simple but slow for deep pages. Keyset/cursor pagination uses the last seen value as a bookmark — O(log n) regardless of page depth.
Partitioning — Dividing data across multiple nodes or tables. Horizontal (sharding): each shard holds a different row range. Vertical: split columns across tables. Partition key choice determines hotspot risk.
Polling — Repeatedly querying a source at intervals for new data. Simple to implement but wastes resources when there's nothing new. Long polling holds the connection open until data arrives, reducing wasted requests.
Polyglot Persistence — Using different databases for different parts of a system, each chosen for its data model strengths. E.g. Postgres for transactional data, Redis for caching, Elasticsearch for search, Cassandra for time-series writes.
Primary Key — A column or set of columns that uniquely identifies each row in a table; must be unique and non-null. The DB creates a B-tree index on it automatically. Natural keys (email) vs surrogate keys (UUID / serial).
Protocol Buffers — Google's language-neutral binary serialization format defined in .proto schema files. 3–10× smaller and faster to parse than JSON. Used by gRPC and internally at Google, Uber, Netflix.
Pub/Sub — Publish/Subscribe: publishers emit events to named topics without knowing who listens; subscribers receive events asynchronously. Decouples producers from consumers. Examples: Google Pub/Sub, Kafka topics, Redis pub/sub.
Q
Query Planner — The database component that enumerates physical execution plans for a SQL query, estimates the cost of each using column statistics, and picks the cheapest. Run EXPLAIN ANALYZE to see the chosen plan and actual vs estimated row counts.
Queue — A FIFO data structure (or distributed messaging system) used for task buffering, async work, and rate smoothing. Decouples producers and consumers so each scales independently. O(1) enqueue and dequeue.
Quorum — The minimum number of nodes that must acknowledge an operation for it to succeed. With N=3 replicas, quorum = 2. R + W > N guarantees at least one node in every read overlaps with the last write, ensuring consistency.
R
Race Condition — A bug where behavior depends on the uncontrolled relative timing of concurrent operations. Classic example: read-modify-write without a lock. Fixed with mutexes, atomic operations, or database-level transactions.
RAG — Retrieval-Augmented Generation: augmenting LLM responses by first retrieving relevant documents from a vector store and injecting them into the prompt. Gives LLMs access to private or recent knowledge without fine-tuning.
Rate Limiting — Capping requests per client per time window to protect a service from abuse or overload. Algorithms: token bucket (allows bursts), fixed window (simple), sliding window (accurate). Return 429 with a Retry-After header.
Read Replica — A DB copy that asynchronously receives writes from the primary and serves reads, increasing read throughput. Introduces eventual consistency — replicas may lag the primary by milliseconds to seconds.
Referential Integrity — Constraint ensuring every foreign key value points to an existing primary key row. Enforced by the DB — prevents orphaned records. CASCADE DELETE automatically removes dependent rows when a parent is deleted.
Replication — Maintaining synchronized copies of data across multiple nodes for fault tolerance and read scaling. Single-leader (most common), multi-leader (multi-region writes), leaderless (Cassandra/Dynamo). Sync replication = durability; async = lower write latency.
REST — Representational State Transfer; stateless HTTP architecture using verbs (GET, POST, PUT, PATCH, DELETE) on noun resources (/users/123). Resources identified by URL; state transferred in JSON or XML. The dominant web API style.
Retry Storm — Many clients retrying simultaneously after a failure, amplifying load on a recovering service and preventing recovery. Mitigated by exponential backoff with jitter and circuit breakers.
Rolling Deployment — Replacing old instances with new ones incrementally, a few at a time, to maintain uptime throughout a deployment. Slower than blue-green but requires only one environment. Kubernetes Deployments use rolling updates by default.
S
Saga Pattern — Managing distributed transactions as a sequence of steps, each with a compensating rollback action if a later step fails. Choreography (events) or orchestration (central coordinator). Avoids distributed locks while ensuring eventual consistency.
Semantic Versioning — Version scheme MAJOR.MINOR.PATCH: breaking change bumps major, new backward-compatible feature bumps minor, bug fix bumps patch. Allows package managers to safely auto-update minor/patch versions.
Serialization — Converting in-memory data to a storable or transmittable byte sequence (JSON, Avro, Protobuf). The reverse is deserialization. Schema evolution (adding/removing fields without breaking old readers) is a key concern in distributed systems.
Service Discovery — The mechanism by which services locate each other dynamically rather than via hardcoded IPs. Client-side (Consul + client library) or server-side (Kubernetes DNS / load balancer). Handles instance churn as pods scale up and down.
Service Mesh — Infrastructure layer (Istio, Linkerd) handling service-to-service mTLS, retries, circuit breaking, load balancing, and telemetry via sidecar proxies — without changing application code.
Sharding — Horizontally partitioning data across multiple DB instances, each holding a distinct subset (a shard). Hash sharding distributes evenly; range sharding enables range queries. Cross-shard joins are expensive or impossible.
Sidecar Pattern — A helper container deployed alongside a main container in the same Kubernetes pod, sharing its network namespace. Handles cross-cutting concerns like logging, proxying, or secrets injection without modifying the main app.
Snowflake ID — A 64-bit distributed unique ID encoding timestamp + worker ID + sequence number. Time-sortable, globally unique, generated without coordination. Invented by Twitter; variants used by Discord, Instagram, and most large platforms.
Single Point of Failure (SPOF) — A component whose failure brings down the entire system. Eliminate by adding redundancy: multiple servers behind a load balancer, read replicas with automatic failover, multi-AZ deployments. Every SPOF is an unacceptable availability risk in any production system targeting 99%+ uptime.
SOLID — Five OOP design principles: Single responsibility (one reason to change), Open/closed (open for extension, closed for modification), Liskov substitution (subtypes must be substitutable), Interface segregation (small focused interfaces), Dependency inversion (depend on abstractions).
SQL Injection — Attack embedding malicious SQL into user input that an application executes unescaped against its database. Prevention: always use parameterized queries or prepared statements — never concatenate user input into SQL strings.
Sticky Session — Load balancer configuration routing all requests from the same client to the same backend server, identified by a cookie or IP. Required for stateful servers that hold session data in memory. Problematic for availability — if that server fails, the user's session is lost. Prefer externalized session state (Redis) so any server can handle any request.
SSE — Server-Sent Events: one-directional HTTP streaming from server to browser over a persistent connection. Simpler than WebSocket for push-only scenarios (live dashboards, notifications). Automatically reconnects on drop.
Stateless — A server storing no per-session client state — any replica can handle any request given the same inputs. Enables horizontal scaling and easy failover. Session state is moved to the client (JWT) or a shared store (Redis).
Stream Processing — Processing data record-by-record in real time as it arrives rather than accumulating it first. Frameworks: Flink, Kafka Streams, Spark Structured Streaming. Enables sub-second latency for alerts, fraud detection, and live dashboards.
T
Tail Latency — The slowest fraction of requests (p95, p99, p999). Often more damaging to UX than average latency because a user waiting on multiple services experiences the worst of each. Track and alert on p99, not the mean.
Technical Debt — The cost of shortcuts taken for speed that must eventually be paid back via refactoring. Like financial debt, it accrues interest: messy code slows every future feature built on top of it.
Throughput — Operations processed per unit time (req/s, rows/s, MB/s). Distinct from latency — a system can have high throughput with high latency (batch pipeline) or low throughput with low latency (interactive API).
Time-to-Live (TTL) — A value specifying when a record (cache entry, DNS record, session token) expires and should be discarded. Short TTL = fresher data, more cache misses. Long TTL = more hits, staleness risk. Add jitter to avoid synchronized expiry storms.
TLS — Transport Layer Security; cryptographic protocol providing confidentiality and integrity over a network connection. Successor to SSL. TLS 1.3 requires 1 RTT handshake vs 2 for TLS 1.2. Used in all HTTPS traffic.
Tombstone — A deletion marker used in distributed databases (Cassandra, DynamoDB, LSM-tree systems) instead of immediately removing data. During reads, tombstones suppress the deleted value. They are garbage-collected during compaction. Accumulation of uncompacted tombstones causes read latency spikes — a common operational issue in Cassandra.
Tokenization — Breaking text into smaller units (tokens) for LLM processing. Roughly 1 token ≈ 0.75 English words. Tokenization boundaries affect prompt cost and context window usage; different models use different tokenizers (BPE, SentencePiece).
Tracing (Distributed) — Tracking a request's path across multiple services using a shared trace ID propagated via headers. Each service adds a span recording its portion of work. Tools: OpenTelemetry, Jaeger, Zipkin. Essential for debugging latency in microservices.
Transaction — A unit of DB work satisfying ACID; either all operations commit or all roll back together. Delimited by BEGIN … COMMIT / ROLLBACK. Keep transactions short to reduce lock contention and connection hold time.
Two-Phase Commit — Distributed protocol where a coordinator first asks all participants to "prepare" (Phase 1) and then commands them to "commit" or "abort" (Phase 2). Guarantees atomicity across nodes but blocks if the coordinator crashes mid-protocol.
U
Union-Find — Also called Disjoint Set Union (DSU); a data structure tracking which elements belong to the same connected component. Union merges two components; Find returns a component's representative. With path compression and union by rank, both operations run in near-O(1) amortized time. Used in Kruskal's MST, cycle detection, and network connectivity problems.
Upstream / Downstream — Upstream = a dependency your service calls; downstream = a service that calls yours. A failure upstream cascades downstream. Always define SLAs for your upstreams and defensive timeouts for calls to them.
Upsert — An insert that becomes an update if the record already exists, avoiding a read-then-write race condition. Postgres: INSERT … ON CONFLICT DO UPDATE. MySQL: INSERT … ON DUPLICATE KEY UPDATE. Idiomatic for idempotent write operations.
UUID — Universally Unique Identifier; 128-bit value generated without a central coordinator. UUIDv4 is random; UUIDv7 is time-sortable (better for DB indexes — avoids page splits from random inserts). Common surrogate key choice.
V
Vector Clock — A timestamp mechanism tracking causality across distributed nodes; each event increments a per-node counter. Allows partial ordering of events and detection of concurrent conflicting writes. Used by Dynamo-style systems for conflict resolution.
Vector Database — Database optimized for storing and similarity-searching high-dimensional embedding vectors at scale. Uses approximate nearest-neighbor indexes (HNSW, IVF) for fast retrieval. Examples: Pinecone, Weaviate, Chroma, pgvector.
Vertical Scaling — Upgrading a single machine with more CPU, RAM, or disk (scale up). Simpler than horizontal scaling but has a hard hardware ceiling and involves downtime for some databases. Databases are often scaled vertically before sharding.
Virtual Machine — Software emulation of a full computer with its own OS kernel, running on a hypervisor (VMware, KVM, Hyper-V). Stronger isolation than containers but heavier (seconds to start, GBs of overhead). EC2 instances are VMs.
W
WAL — Write-Ahead Log: all changes are written sequentially to an append-only log before being applied to the main data files. Enables crash recovery (replay the log) and replication (stream the log to replicas). Used by PostgreSQL, MySQL, RocksDB.
Webhook — Server-to-server HTTP callback triggered by an event; the receiver acts on data pushed to it rather than polling. The sender retries on failure. Secure by verifying an HMAC signature on the payload.
WebSocket — Persistent full-duplex TCP connection established via an HTTP upgrade request. Enables real-time bidirectional browser-server communication. Used for chat, live collaboration, gaming, and financial tickers.
Worker Thread — A separate thread for offloading CPU-intensive work so the main thread (or event loop) stays responsive. In Node.js, the worker_threads module runs compute tasks (image processing, crypto, JSON parsing of huge payloads) without blocking I/O. In browsers, Web Workers enable background computation without freezing the UI.
Write-Behind Cache — Writes update the cache immediately and the database asynchronously later. Higher write performance but risk of data loss if the cache crashes before the DB is synced. Contrast with write-through (synchronous DB write).
Write-Through Cache — Writes update cache and database simultaneously in the same request. Reads are always fast and the cache is never stale immediately after a write. Latency is that of the slower DB write.
X
XSS — Cross-Site Scripting; injecting malicious scripts into a web page viewed by other users to steal session tokens or redirect them. Three types: stored (persisted in DB), reflected (in URL), DOM-based (manipulates client-side JS). Prevented by output encoding and Content Security Policy headers.
Y
YAGNI — You Aren't Gonna Need It: don't implement something until it's actually required. One of the core Extreme Programming principles. Prevents over-engineering that adds complexity without current value.
Z
Zero Downtime Deployment — Deploying a new version while keeping the service continuously available via blue-green, canary, or rolling update strategies. Requires backward-compatible DB migrations that run before the new code is activated.
ZooKeeper — Distributed coordination service providing leader election, distributed locks, config management, and service discovery via a strongly consistent, replicated file-system-like API. Used by Kafka (historically), HBase, and many older distributed systems.
